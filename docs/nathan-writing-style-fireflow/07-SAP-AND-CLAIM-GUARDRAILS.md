# SAP and Claim Guardrails

## Core rule

FireFlow may demonstrate understanding of SAP SD aligned order-to-cash concepts. It must not imply professional SAP use that Nathan cannot defend.

## Approved language

- SAP SD aligned workflow concepts
- order-to-cash operating model
- designed around common order entry, pricing, delivery, billing, and customer master dependencies
- demonstrates my understanding of the operational sequence
- shows how I would approach learning and supporting the company's SAP environment
- independent front-end concept
- synthetic order and case data
- common SAP document reference

## Use carefully

- SAP informed
- SAP readiness
- transaction codes
- enterprise workflow

These terms need nearby context explaining what is conceptual, synthetic, or learned rather than professionally operated.

## Prohibited language

- SAP expert
- eight years of SAP experience
- built in SAP
- integrated with SAP
- implemented SAP SD
- production SAP solution
- used Samyang's SAP data
- improved Samyang fill rates
- resolved real Samyang deductions

## Transaction codes

If transaction codes are shown, label them as common references or glossary terms. Do not make them the main evidence of expertise.

The main evidence should be Nathan's understanding of:

- required information
- order dependencies
- exception visibility
- ownership
- customer communication
- pricing mismatches
- delivery commitments
- billing disputes
- deductions
- root cause prevention

## Data claims

Do not invent customers, orders, employees, service levels, fill rates, CSAT, revenue, sales, or outcomes. Synthetic data must remain labeled wherever it appears.
